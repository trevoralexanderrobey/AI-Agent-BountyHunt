# Burp Suite Pro Integration (BionicLink)

This integration adds a local “nervous system” bridge between:
- Burp Suite Pro (traffic + tooling)
- BionicLink (Burp extension HTTP server on loopback)
- OpenClaw Bridge Service (queue + tool runner + summarization)
- OpenClaw skills (call bridge `/execute-tool`)

## Components

1) **BionicLink (Java Burp extension)**
- Runs inside Burp
- Binds to `127.0.0.1`
- Default port: `8090`
- Exposes minimal endpoints used by the bridge

2) **OpenClaw Bridge Service (Node.js)**
- Local daemon on `127.0.0.1:8787`
- Enforces:
  - **Scope Lock**: blocks Repeater/Scan unless Burp considers the URL in scope
  - **Active Scan Guard**: blocks `burp_active_scan` unless `BURP_ALLOW_ACTIVE_SCAN=true`
  - **TSP summarization**: prunes headers and truncates bodies before returning data to an AI agent

3) **OpenClaw Skill: `burp-suite`**
- Lives at `~/.openclaw/skills/burp-suite/`
- Calls the bridge’s `POST /execute-tool`

## Burp Tool Calls (via Bridge)

The bridge supports a Burp “tool mode” on `POST /execute-tool` when `skill` is omitted.

Allowlisted Burp tools:
- `burp_get_history`
- `burp_analyze_request`
- `burp_active_scan` (guarded)

Examples:

```bash
curl --fail --silent --show-error \
  -H "Content-Type: application/json" \
  -X POST \
  https://127.0.0.1:8787/execute-tool \
  -d '{"tool":"burp_get_history","args":{"limit":20,"inScope":true}}'
```

## Scope Lock (Required for Repeater + Scan)

The bridge will refuse `burp_analyze_request` or `burp_active_scan` unless:
1. The URL is within Burp’s Target Scope, and
2. BionicLink confirms it via `GET /scope?url=...`

If you see `out of scope` errors:
- add the host/URL pattern to Burp’s scope
- retry the tool call

## Traffic Summarization Protocol (TSP)

Burp history/repeater responses can be very large. The bridge applies summarization so agents receive:
- a compact list of request lines (host/path/status/len)
- minimal token-safe derived hints (e.g., cookie names)
- short `body_preview` fields (truncated and normalized)

This is implemented in:
- `openclaw-bridge/bridge/tsp.ts`

## Protocol Stability Ingest (`POST /bionic-ingest`)

The bridge also supports ingesting captured HTTP request/response pairs for defensive stability analysis.

Endpoint:
- `POST https://127.0.0.1:8787/bionic-ingest`

What it does:
- Normalizes + redacts common secret headers (`Authorization`, `Cookie`, `Set-Cookie`, etc).
- Writes the capture to the job workspace as `BIONIC_INGEST_EVENT.json`.
- Creates a standard bridge job so `MISSION_REPORT.md` is generated by OpenClaw using a stability-focused QA persona.

You can manually send a capture (example):

```bash
curl --fail --silent --show-error \
  -H "Content-Type: application/json" \
  -X POST \
  https://127.0.0.1:8787/bionic-ingest \
  -d '{
    "url": "https://example.com/api/submit",
    "method": "POST",
    "request_headers": [{ "name": "Content-Type", "value": "application/json" }],
    "request_body": "{\"hello\":\"world\"}",
    "status": 200,
    "response_headers": [{ "name": "Content-Type", "value": "application/json" }],
    "response_body": "{\"ok\":true}"
  }'
```

## Active Scan Guard (High Noise)

Active scan is blocked by default.

To enable (explicit opt-in):
1. Edit: `openclaw-bridge/.env`
2. Set: `BURP_ALLOW_ACTIVE_SCAN=true`
3. Restart:

```bash
npm run bridge:restart
```

## End-to-End Flow (Skill)

The OpenClaw skill tools call the bridge:
- `~/.openclaw/skills/burp-suite/tools.js`

In Burp mode, the skill does not need to pass `skill` in the payload; it simply calls:
- `POST https://127.0.0.1:8787/execute-tool`

## Troubleshooting

If `curl https://127.0.0.1:8090/health` fails:
- confirm the jar is loaded in Burp
- confirm Burp is running
- confirm the port (default `8090`) is not in use

If bridge health reports BionicLink unreachable:
- verify `BIONICLINK_BASE_URL` in `openclaw-bridge/.env`
- restart the bridge
